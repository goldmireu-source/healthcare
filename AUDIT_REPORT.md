## 요약

이 프로젝트는 "FastAPI 미니 과제"로 시작해 지금은 인증, 권한분리, 어뷰징 방지,
다크모드 반응형 UI, 커스텀 데이터시각화, 그리고 "AI Health Coach"라는 이름의
분석/코칭 레이어까지 갖춘 상당히 완성도 높은 개인 프로젝트입니다. 구조적으로도
Provider 패턴·Service Layer 분리 등 실무 수준의 설계 판단이 곳곳에 보입니다.

다만 **"AI Health Coach"는 현재 이름과 달리 실제 LLM을 호출하지 않는 규칙 기반
로직**입니다. 대신 `CoachingProvider`라는 교체 가능한 인터페이스로 미리 설계되어
있어서, 준비하신 ChatGPT API 키를 연결하기에 이상적인 상태입니다 — 이 부분을
4번(AI 접목 제안)에서 구체적으로 다룹니다.

## 1. 잘된 점

- **인증/권한 설계가 견고함**: 비밀번호 PBKDF2 해시, DB 기반 세션(JWT 아님 → 즉시
  폐기 가능), 계정 잠금(5회 실패시 423), IP 기준 rate limit, IDOR 방지(모든 기록
  조회가 `user_id == current_user.id`로 스코프됨, 직접 코드로 확인함)까지 개인
  프로젝트치고 이례적으로 꼼꼼합니다.
- **관리자/유저 페이지의 완전한 분리**: 서로를 참조하는 링크·리다이렉트가 코드상
  하나도 없고, 각자 독립된 로그인을 가짐 — 실제 서비스의 admin 패널 설계와 유사한
  방식으로, 권한 경계가 프론트엔드 실수로 무너질 여지를 원천 차단합니다.
- **AI 코칭 레이어의 확장 설계**: `health_coach.py`의 `CoachingProvider(ABC)` +
  `integrations.py`에 `OpenAICoachingProvider` 연동 예시 코드가 주석으로 미리
  문서화되어 있음. 실제 LLM을 붙일 때 호출부(main.py)나 응답 스키마를 안 건드리고
  provider 클래스 하나만 추가하면 되는 구조 — 이런 "나중에 실제로 API 키를 넣을
  사람"을 배려한 설계는 흔치 않습니다.
- **Service Layer 분리를 검증까지 하며 진행**: `health_service.py` 등으로 로직을
  옮기면서 "분리 전/후 응답을 diff해서 완전히 동일함을 확인"했다고 기록돼 있음 —
  리팩터링 중 가장 위험한 회귀를 스스로 견제한 흔적.
- **데이터시각화가 라이브러리 없이 SVG 직접 그리기 + 접근성 고려**(라벨은 innerHTML
  대신 textContent로 렌더링해 XSS 방지)로 되어 있고, 다크모드/반응형 3단까지
  Playwright로 실제 스크린샷 검증을 거침.
- **문서화 습관**: `PROJECT_CONTEXT.md`가 기능 추가마다 배경·검증 방법·발견한 버그까지
  꾸준히 기록되어 있어, 코드만 봐서는 알기 어려운 "왜 이렇게 했는지"가 남아있습니다.
  개인 프로젝트에서 이 정도 문서화는 드뭅니다.

## 2. 미비한 점 (구조적 한계)

- **자동화 테스트가 전혀 없음**. 지금까지의 검증은 전부 curl 수동 실행 + 1회성
  Playwright 스크립트(코드로 남지 않음)입니다. 기능이 37개 엔드포인트로 늘어난
  지금, 뭔가 하나 고칠 때마다 수동으로 회귀 확인하는 비용이 기하급수적으로
  커지고 있을 시점입니다.
- **DB 마이그레이션 도구(Alembic 등)가 없음**. 지금까지 컬럼이 추가될 때마다
  `data/health_log.db`를 통째로 지우고 재생성하는 방식으로 해결해왔습니다. 로컬
  개발 단계에선 문제없지만, **실서버에 배포해 실사용자 데이터가 쌓이기 시작하면
  이 방식은 쓸 수 없습니다.**
- **rate limiter가 인메모리**라 (1) 서버 재시작 시 초기화, (2) uvicorn을 워커
  여러 개로 띄우면 워커별로 카운터가 따로 놀아 사실상 무력화됩니다. 코드 주석에
  한계가 명시돼 있어 인지는 하고 있는 상태.
- **CORS 미들웨어·보안 헤더(HSTS/CSP/X-Frame-Options) 설정이 코드에 없음**
  (grep으로 직접 확인). 지금은 same-origin 정적 서빙이라 당장 문제는 아니지만,
  배포 시 점검 목록에 넣어야 합니다.
- **프론트엔드 코드 중복**: `index.html`과 `admin.html`이 같은 디자인 토큰/CSS를
  각자 파일에 복붙해서 갖고 있음. 하나 고치면 둘 다 손으로 맞춰야 합니다.
- **CSV Import는 "미리보기"만 하고 실제 DB 저장에 연결되지 않음** (문서에도
  명시된 후속 작업).

## 3. 보완할 수 있는 점

우선순위가 높은 순서대로:

1. **Alembic 도입** — 지금 방식(DB 통째로 삭제 후 재생성)은 배포 전 반드시
   손봐야 할 부분. 실사용자 데이터가 생기기 전, 지금이 도입하기 가장 쉬운
   시점입니다.
2. **pytest 기반 테스트 스위트 구축** — 지금까지 기능 추가 때마다 반복해온 curl
   시나리오(로그인 실패 5회→잠금, IDOR, cascade 삭제 등)를 그대로 테스트 코드로
   옮기기만 해도 회귀 방지 효과가 큽니다. `TestClient` + SQLite 임시 파일 조합이면
   지금 구조를 거의 안 건드리고 도입 가능.
3. **rate limiter를 Redis 등 외부 저장소로 전환** — Lightsail에 올리는 시점에
   워커를 2개 이상 쓸 가능성이 있다면 필수. 워커 1개로 유지할 거라면 당장은
   위험도가 낮아 후순위로 미뤄도 됨.
4. **CORS/보안 헤더 명시적 설정** — `fastapi.middleware.cors.CORSMiddleware` +
   `secure` 계열 헤더 미들웨어. 배포 체크리스트에 `COOKIE_SECURE=true`와 함께
   묶어서 관리.
5. **프론트엔드 디자인 토큰 공유화** — 최소한 CSS 변수 블록만이라도 별도
   `theme.css`로 분리해 두 페이지가 import하게 하면 중복 관리 부담이 줄어듦
   (빌드 도구 없이도 `<link>` 태그로 가능).

  대응이나 보안 사고 조사에 유용
- **다국어(i18n)** — 지금은 전부 한국어 하드코딩이라, 필요해지면 초기에 문자열을
  분리해두는 게 나중보다 훨씬 쉬움

## 5. 취약점

### 5-1. CSV Export 수식 인젝션 (Formula Injection) — 실제 코드에서 확인됨

`GET /export/csv`가 `memo` 필드(최대 500자 자유 입력)를 아무 이스케이프 없이
CSV 셀에 그대로 씀. `memo`가 `=`, `+`, `-`, `@`로 시작하면 엑셀/구글시트에서
**수식으로 해석되어 실행될 수 있는** 잘 알려진 취약점(CSV Injection)입니다.

- 지금은 본인 데이터만 export하므로 "자기 자신을 공격"하는 셈이라 실질 피해는
  제한적입니다.
- 하지만 **관리자가 특정 사용자 데이터를 export하는 기능**(자연스러운 다음 확장)이
  생기는 순간, 악의적 사용자가 memo에 심어둔 수식을 관리자가 열어보다 실행하게
  되는 실질적 공격 경로가 됩니다.
- **조치**: 셀 값이 `=+-@`로 시작하면 앞에 작은따옴표(`'`)를 붙이는 표준 방어
  코드 한 줄이면 해결됨. 지금 고쳐두는 게 비용이 가장 쌉니다.

### 5-2. 세션 만료 정리(cleanup) 로직 부재

`Session` 테이블에서 만료된 row를 자동으로 지우는 코드가 안 보였습니다
(로그아웃 시 명시적으로 지우는 `delete_session`만 존재). 로그아웃 안 하고 방치된
세션은 테이블에 계속 쌓입니다. 보안 취약점이라기보다 운영 이슈(테이블 비대화)에
가깝지만, 배포 전에 스케줄 정리(주기적 DELETE WHERE expires_at < now)를
추가하는 걸 권장합니다.

### 5-3. 이미 인지되어 있는 항목 (재확인)

- rate limiter의 멀티 워커 무력화 (2번 참고)
- 보안질문 기반 비밀번호 찾기는 이메일 인증보다 구조적으로 약함(생일·색깔처럼
  추측 가능한 답 유형) — rate limit(IP+아이디 5회/15분)로 완화하고 있으나 근본
  해결은 아님
- HTTPS/`COOKIE_SECURE` 미설정 상태로 배포하면 세션 토큰이 평문 HTTP로 노출됨
  (문서에 배포 시 조치 필요로 이미 기록돼 있음)

## 6. AI 접목 가능성 (ChatGPT API 활용 제안)

**결론부터: 코드가 이미 이 순간을 위해 설계돼 있습니다.** `health_coach.py`의
`CoachingProvider` 추상 클래스와, `integrations.py`에 남겨진 아래 예시 코드가
그대로 청사진입니다:

```python
class OpenAICoachingProvider(health_coach.CoachingProvider):
    def __init__(self, api_key: str, model: str = "gpt-4o-mini"):
        self.api_key = api_key
        self.model = model

    def generate(self, records, goal=None):
        trends = analyze_trends(records)      # 기존 분석 결과를 프롬프트 재료로 재사용
        prompt = _build_prompt(trends, goal)
        response = call_openai_api(self.api_key, self.model, prompt)
        return _parse_bullet_messages(response)
```

### 구체적으로 연결하기 좋은 지점 (우선순위순)

1. **`GET /health-coaching` 실제 LLM화** — 지금은 규칙 5개가 조건에 맞으면
   고정 문장을 반환. `analyze_trends()`가 이미 만들어주는 지표별 추세를 프롬프트에
   넣고, ChatGPT가 자연어로 더 개인화된 조언을 생성하도록 교체. 실패 시 지금의
   규칙 기반 로직으로 자동 폴백(현재 구조가 이미 이 폴백에 유리함 — provider를
   None으로 넘기면 기본 규칙 기반이 실행됨).
2. **`generate_weekly_summary()` 자연어 고도화** — 지금은 어간+연결어 조합으로
   문장을 짜맞추는 방식(문법은 맞지만 표현이 제한적). 같은 입력(this_week/
   last_week/change dict)을 프롬프트로 던지면 훨씬 풍부한 문단이 가능.
3. **건강검진 PDF 파싱** — `MockHealthCheckupPdfImporter`를 비전 모델(PDF를
   이미지로 변환 후 API에 전달) 또는 텍스트 추출+LLM 파싱으로 교체.
4. **관리자용 이상 징후 자연어 요약** — `risk_detection.py`가 만드는 LOW/MEDIUM/
   HIGH 판정 결과를, 관리자 개요 화면에서 "이번 주 위험 사용자 3명의 공통 패턴은
   ~"처럼 자연어로 요약해주면 운영 관점에서 유용.

### 실제로 연결할 때 챙길 것

- **API 키는 서버 환경변수로만** (예: `OPENAI_API_KEY`), 프론트엔드 코드에는
  절대 노출 금지 — 지금 구조(FastAPI가 provider를 서버에서 인스턴스화)라면
  자연스럽게 지켜지지만, 실제 구현 시 재확인 필요.
- **비용/지연 관리** — 매 조회마다 호출하면 비용이 쌓이므로, 하루 1회 캐싱
  (예: 사용자당 오늘 이미 생성한 코칭 메시지는 재사용) 권장. `badges.py`가
  이미 쓰고 있는 "조회 시점에 계산 후 저장하는 지연 평가" 패턴을 그대로
  재사용할 수 있음.
- **개인 건강정보를 외부 API로 전송하는 것에 대한 고지** — 체중·혈압·혈당 같은
  민감정보가 OpenAI로 나간다는 걸 사용자에게 알리는 최소한의 안내(회원가입 시
  또는 코칭 기능 최초 사용 시 동의) 필요. 학습용 프로젝트라도 습관을 들여두면
  좋은 부분.
- **실패 처리** — API 타임아웃/오류 시 빈 화면이 아니라 지금의 규칙 기반
  메시지로 자연스럽게 대체되도록(이미 인터페이스가 그렇게 설계돼 있어 구현
  난이도는 낮음).
