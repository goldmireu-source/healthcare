## 1. 프로젝트 구조 분석

**폴더 구조**: `backend/`(FastAPI 21개 모듈) / `frontend/static/`(index.html, admin.html,
theme.css) / 루트에 Dockerfile·문서. 여러 작업자가 나눠 맡는 상황을 가정해 물리적으로
분리했다고 문서화되어 있고, 실제로 백엔드 파일이 라우트(`main.py`)-서비스(`*_service.py`)
-모델(`models.py`)-스키마(`schemas.py`)로 책임이 나뉘어 있음.

**레이어 분리**: 서비스 레이어(`health_service.py`, `goal_service.py`,
`report_service.py`, `admin_service.py`)가 실제로 존재하고 `main.py`가 이를 호출하는
구조는 맞음. 다만 **`main.py`가 1,125줄로, 40개 라우트 정의 자체가 전부 이 한 파일에
있음** — 서비스 레이어 분리는 됐지만 라우트 등록은 `APIRouter`로 도메인별
분리(`auth_router`, `records_router` 등)가 되어있지 않음. FastAPI 프로젝트에서 흔히
쓰는 패턴이 빠져 있어, 지금 규모(40개 엔드포인트)에서 이미 탐색성이 떨어지기 시작하는
지점.

**모듈화 수준**: AI Health Coach 관련 로직(`health_coach.py`, `health_trends.py`,
`risk_detection.py`, `health_score.py`, `goal_prediction.py`, `health_calendar.py`,
`health_timeline.py`, `badges.py`)이 기능 단위로 잘게 분리되어 있고, 전부 `main.py`
또는 서비스 레이어에서 실제로 import되어 사용됨 — **dead code(고아 모듈) 없음을
grep으로 직접 확인**.

**유지보수성**: 백엔드는 위와 같이 양호. 반면 **프론트엔드는 `index.html`이 자체
`apiGet`/`apiSend` 공통 헬퍼(401 처리 포함)를 쓰는데, `admin.html`은 `fetch()`를
매 함수마다 직접 호출**하고 있음(grep으로 확인 — admin.html에 `apiGet`/`apiSend`/
`checkAuth` 함수가 아예 없음). CSS는 `theme.css`로 통합했지만 JS 공통 로직은 통합
안 된 상태라, 인증 처리 로직을 고칠 일이 생기면 두 파일에 따로 반영해야 함.

**확장성**: `CoachingProvider`(AI), `WearableDataSource`/`HealthDataImporter`(외부
연동) 등 인터페이스 기반 확장 지점이 실제로 설계돼 있고, OpenAI 연동이 그 설계
그대로 들어간 것으로 이미 검증됨(README/PROJECT_CONTEXT 근거).

**중복 코드**: 프론트엔드 두 파일 사이의 인증/토스트/모달 관련 JS가 사실상 중복
구현. `toast()` 함수가 파일마다 각자 정의(정상적인 범위의 중복). 백엔드 쪽 중복은
발견하지 못함.

**Dead Code / 미사용 파일**: 없음 (전 모듈 import 사용처 확인 완료). 미사용 npm
패키지 같은 개념은 해당 없음(Node 프로젝트 아님).

**Legacy 코드**: 없음 — 프로젝트 시작이 최근이라 레거시라 부를 코드가 아직 없음.

**평가**: **좋음** (백엔드 레이어 분리·확장성은 우수, `main.py` 비대화와 프론트 JS
중복이 "매우 좋음"을 막는 감점 요인)

| 영역 | 기능 |

| 목표/리포트 | 목표 설정+달성여부, 목표달성 예측일, 주간 리포트(비교+AI 자연어 요약) |
| AI Health Coach | 코칭 메시지(OpenAI 연동+규칙기반 폴백+일 1회 캐싱), 추세분석, 이상징후감지, Health Score, 건강캘린더, 타임라인, 배지 |

| 관리자 | 사용자 관리(검색/필터/정렬/페이지네이션), 전체 통계+Analytics(활동률/유지율/전환율), 사용자별 기록조회, 회원정보 상세모달, 계정삭제/강제로그아웃, 감사로그, 접속중 표시(근사치), KPI 드릴다운 |
| 프론트 공통 | 다크모드, 반응형 3단, 로딩/빈상태/에러 상태 처리, 커스텀 토스트/모달 |

### 추가 제안 기능과 근거

| 제안 기능 | 왜 필요한가 |
|---|---|
| **알림/리마인더(웹 푸시)** | 지금은 사용자가 스스로 접속해야만 기록을 남김. 습관형 서비스(건강기록)는 리텐션이 곧 핵심 지표인데, 능동적 상기 장치가 전무해 실사용 시 3일차 이탈이 클 것으로 예상됨(정량 데이터는 없음, 서비스 카테고리 특성상 일반론). |
| **소셜/공유 최소 기능(가족과 공유)** | 건강관리는 실제로 가족 단위 관심사인 경우가 많음(부모 혈압 체크 등). "다른 사용자와의 공유"가 Out of Scope로 명시돼 있는데, 결제/소셜 로그인 같은 무거운 기능이 아니라 "보기 전용 공유 링크" 정도의 최소 기능은 차별화 포인트가 될 수 있음. |
| **웨어러블 실연동(우선 Google Fit/Apple Health 중 1개)** | 매일 손으로 입력하는 것 자체가 이탈 요인. 인터페이스는 이미 준비돼 있어 상대적으로 착수 비용이 낮음. |
| **온보딩 튜토리얼/샘플 데이터 체험** | 가입 직후 빈 화면(뒤 4번 UX 분석 참고)이 첫인상을 해칠 가능성이 있어, 첫 방문자에게 무엇을 하면 되는지 알려주는 장치가 필요. |
| **PWA(홈 화면 추가, 오프라인 캐시 일부)** | 매일 쓰는 서비스 특성상 브라우저 탭보다 홈 화면 아이콘 진입이 재방문률에 유의미한 차이를 만듦. 지금 구조(빌드 없는 정적 HTML)에서 manifest.json + service worker 추가는 비교적 저비용. |
| **관리자 화면 데이터 시각화 확장(사용자 페이지 수준)** | 유저 페이지는 라인차트·막대그래프가 있는데 관리자 개요는 가입추이 막대그래프 1종뿐. 운영 의사결정에 쓸 지표(리텐션 코호트 등)가 시각화되어 있지 않음. |

**평가 기준별**: 사용성(현재 기능 자체의 사용성은 3번에서 별도 평가) / 실용성 —
건강 기록·분석 기능 자체는 실사용 가치가 명확함 / 완성도 — 백엔드 40개 엔드포인트
기준으로는 매우 높음, 다만 알림 등 "재방문을 만드는 장치"가 없어 완결성엔 구멍 /
차별성 — AI 코칭+관리자 Analytics 결합은 개인 프로젝트치고 드문 조합 / 시장 경쟁력 —
14번 항목에서 별도 상세.

## 3. UI 분석

**사용성/정보구조**: 유저 페이지가 "기록 / 통계&리포트 / 목표" 3개 뷰로 나뉜 사이드바
구조, 관리자는 "개요 / 사용자관리 / 감사로그" — 정보 구조 자체는 단순하고 명확함.
대시보드 히어로(건강 스코어 링 + 지표 카드)가 최상단에 있어 "오늘 내 상태"를 한눈에
보여주는 정보 위계가 적절함.

**컬러**: `theme.css` 기준 브랜드 teal(#0D9488)과 상태색(`--status-good` #17A34A)을
명확히 분리해 놓은 점이 특히 좋음 — 초록 계열 브랜드색이 "정상"으로 오독되는 흔한
실수를 피함. 경고(brick, #DC2626)·주의(amber, #D97706)까지 3단 상태색 체계가
전역에서 일관되게 재사용됨.

**타이포그래피**: Pretendard Variable(본문) + IBM Plex Mono(수치) 조합은 데이터
중심 서비스에 적절한 선택. 다만 실제 코드를 세어보면 `index.html` 기준 12~13px
텍스트가 여전히 35곳— 최근 "텍스트 크기 확대" 커밋 이후에도 잔존. 안내문/보조 텍스트
용도로는 무리 없으나, 본문성 텍스트에도 이 크기가 쓰인 곳이 있다면 가독성 저하 요인.

**버튼/아이콘**: 사이드바 네비게이션에 인라인 SVG 아이콘 도입까지는 했으나, 아이콘
단독 버튼(수정/삭제 등)에 접근성 라벨이 없음(8번 UX/접근성에서 상세).

**여백/Radius/Shadow**: `--radius-sm/md/lg`(8/12/16px), `--shadow-sm/md` 토큰이
일관 적용됨 — 디자인 시스템으로서 갖출 건 갖춤.

**반응형**: 데스크톱(사이드바) / 태블릿 721~1024px(아이콘 레일) / 모바일 ~720px(하단
탭바) 3단 대응이 실제 CSS 미디어쿼리로 구현돼 있고, Playwright로 실브라우저
스크린샷 검증까지 거쳤다고 문서화됨(이 세션에서 직접 재현은 못 했음 — **확인 불가**,
코드상 미디어쿼리 존재만 확인).

**접근성**: `aria-*`/`role`/`alt` 속성이 `index.html` 0건, `admin.html` 1건 —
시각 디자인 완성도에 비해 접근성은 사실상 미착수 상태. 색상만으로 위험도를 구분하는
배지(위험/주의/정상)에 텍스트 라벨이 병기되어 있는지는 확인함(병기됨, 색맹 사용자
배려는 되어 있음) — 그러나 스크린리더용 시맨틱 마크업은 별개 문제로 남아있음.

**유명 서비스 대비 격차**:
- **Linear/Notion 대비**: 커맨드팔레트, 키보드 단축키, 스켈레톤 로딩(현재는 "불러오는
  중..." 텍스트) 부재. 마이크로 인터랙션(hover 트랜지션 디테일)은 코드상 일부
  존재(`transition` 속성 사용 확인)하나 전면적이지 않음.
- **Apple Health 대비**: 이 프로젝트의 "건강 스코어 링" UI가 Apple Health의 활동
  링을 참고했다고 문서화돼 있음 — 개념은 유사하나 애니메이션(링이 채워지는 모션)은
  정적 SVG로 구현되어 있어 다이나믹함에서 차이.
- **Material Design 대비**: elevation(그림자 단계), ripple 등 표준 상호작용
  패턴은 미적용. 다만 이 프로젝트가 Material을 표방한 적은 없어 감점 요인이라기보다
  "참고 시 격차" 정도로 이해해야 함.

## 4. UX 분석

**사용자 플로우**: 회원가입(아이디+비밀번호+보안질문 선택) → 로그인 → 기록 입력이
기본 흐름. 단계 수는 적정하나, **가입 직후 첫 화면이 빈 상태**(기록 0건)라는 점을
`empty-note` 문구("아직 기록이 없습니다. 위에서 첫 기록을 추가해보세요")로 안내하고
있음 — 완전히 방치하진 않았으나, 이 이상의 온보딩(샘플 데이터 미리보기, 첫 기록
입력 유도 애니메이션 등)은 없음.

**로딩 경험**: 기록/통계/목표/주간리포트 각 영역에 "불러오는 중..." 텍스트 표시가
코드로 확인됨 — 스켈레톤 UI는 아니지만 최소한의 로딩 피드백은 있음.

**빈 화면 처리**: 기록 없음, 통계 데이터 없음, 목표 없음, 최근 2주 리포트 없음 각각
안내 문구가 개별적으로 존재함(grep으로 5곳 이상 확인) — 이 부분은 잘 되어 있는 편.

**에러 처리/피드백**: 성공/실패를 토스트로 일관 처리(alert/confirm 미사용, 이전
세션에서 이미 확인). API 에러 시 `{"detail": "..."}` 형식이 백엔드 전역에서
일관되므로 프론트가 이를 그대로 토스트에 표시하는 패턴이 성립함.

**회원가입/로그인**: 보안질문 방식이 이메일 인증보다 마찰이 적다는 장점은 있으나,
"보안질문 답을 잊으면 복구 수단이 아예 없다"는 구조적 리스크가 있음(이메일이 없으므로
관리자가 대신 확인해줄 방법도 제한적 — 관리자가 계정을 삭제하거나 새로 만들게
안내하는 것 외엔 없음, 코드상 관리자용 "비밀번호 강제 리셋" 기능 없음을 확인).

**검색/탐색**: 유저 페이지는 날짜범위 검색만 있고 메모 텍스트 검색은 없음. 관리자
페이지는 아이디 검색만 있고(f761bd1 커밋으로 실명 검색 추가됨) 기록 내용 검색은
없음 — 범위가 작아 당장 문제는 아니나 데이터가 쌓이면 아쉬워질 지점.

**이탈 가능성이 높은 구간 (추정, 코드 근거)**:
1. **비밀번호 찾기 실패 시** — 보안질문 답이 틀리면(rate limit까지 걸리면) 복구
   경로가 없어 그대로 이탈할 가능성. (코드로 "복구 불가 시 대안 안내 문구" 유무를
   확인함 — 별도 안내 없음)
2. **가입 직후 첫 기록 입력 전** — 온보딩 장치 부재로, 무엇을 입력해야 할지 감이
   안 잡히는 사용자는 이탈 가능(추정, 실사용 데이터 없음).
3. **모바일에서 긴 입력 폼** — 기록 입력 폼 필드 수가 많아(9개 필드) 모바일 하단
   탭바 환경에서 스크롤이 길어질 가능성(정확한 필드 순서/그룹핑은 코드 확인함,
   실제 모바일 체감은 **확인 불가**).

## 5. 디자인 시스템 분석

- **색상 일관성**: 위 3번 참고 — 브랜드/상태 분리, 라이트·다크 양쪽 정의. **일관성
  높음.**
- **Spacing**: 고정된 spacing 스케일(예: 4/8/12/16/24px 토큰)이 명시적으로 변수화되어
  있지는 않음 — 하드코딩된 px 값이 CSS 곳곳에 직접 쓰임(grep 결과 `padding:`,
  `gap:` 값이 토큰이 아닌 리터럴). 색상/radius/shadow는 토큰화했는데 spacing만
  누락된 상태.
- **Radius/Shadow**: 토큰화 완료(`--radius-sm/md/lg`, `--shadow-sm/md`).
- **Typography**: 폰트 패밀리는 토큰화(변수는 아니지만 전역 선언), 크기는 리터럴
  px 직접 사용 — 타이포 스케일(예: `--text-xs/sm/base/lg`) 부재.
- **Component 재사용성**: 빌드 도구가 없는 순수 HTML/CSS/JS라 컴포넌트 개념 자체가
  없음(React 등의 컴포넌트 재사용과 다른 이야기). 카드/배지/버튼류의 CSS 클래스는
  재사용되고 있으나 JS 로직 재사용은 파일 간에는 안 됨(1번 항목 참고).
- **디자인 토큰 적용 여부**: 색상·radius·shadow는 적용, spacing·typography 스케일은
  미적용 — **부분 적용** 상태.
- **Tailwind 사용 방식**: 미사용(해당 없음). Vanilla CSS + CSS 커스텀 프로퍼티 방식.
- **공통 컴포넌트 구조**: 없음(1번 항목 프론트엔드 중복 이슈와 동일 맥락).

**평가**: **보통** (색상 토큰 체계는 수준급이나 spacing/typography 토큰화와 JS
컴포넌트 재사용 부재가 확장 시 발목을 잡을 것)

## 6. 코드 품질

- **가독성**: 변수/함수명이 한국어 주석 + 영어 코드로 명확함(`_sanitize_csv_cell`,
  `evaluate_record` 등 의도가 드러나는 네이밍).
- **함수 길이**: 대부분 적정. `main.py`의 각 라우트 함수 자체는 짧고(서비스 레이어에
  위임), 문제는 함수 길이가 아니라 **파일 길이**(1번 참고).
- **책임 분리(SRP)**: 서비스 레이어 분리로 상당 부분 지켜짐. `schemas.py`가
  448줄로 큰 편이나, Pydantic 모델 정의 특성상 자연스러운 길이.
- **SOLID/Clean Code**: `CoachingProvider`(OCP를 의식한 확장 지점), 의존성 주입
  패턴(`Depends(get_db)`, `Depends(auth.get_current_user)`)이 FastAPI 관용구를
  잘 따름.
- **중복**: 백엔드는 낮음. 프론트엔드 JS는 앞서 언급한 대로 파일 간 중복 있음.
- **불필요한 State**: 프론트엔드 전역 변수 사용 패턴(React 등 상태관리 라이브러리
  없이 순수 JS 전역변수 + DOM 직접 조작)은 지금 규모(뷰 3~4개)에선 관리 가능한
  수준. 이보다 화면이 늘어나면 전역 상태 추적이 급격히 어려워질 구조.
- **복잡도**: `main.py`가 40개 라우트를 한 파일에 담아 인지 복잡도가 가장 높음.
- **Pydantic Deprecation 경고**: `pytest` 실행 시 54개 경고 발생 — 원인은
  `class Config` 방식(Pydantic V1 스타일)을 V2의 `ConfigDict` 대신 계속 사용하고
  있기 때문(**실행해서 직접 확인**). 지금은 동작하지만 Pydantic V3에서 제거 예정
  기능이라 방치하면 향후 업그레이드 시 전부 깨짐.

**리팩토링 우선순위**:
1. `main.py`를 `APIRouter`로 도메인별 분리(auth/records/goals/admin/ai/integrations)
2. Pydantic `class Config` → `ConfigDict` 전환 (54개 경고 해소)
3. 프론트 JS 공통 로직(apiGet/apiSend/toast/checkAuth)을 `shared.js`로 추출
4. CSS spacing/typography 토큰화

## 7. 성능 분석

- **불필요한 렌더링**: React 등 가상DOM 프레임워크가 아니라 "불필요한 리렌더링"
  개념 자체가 적용되지 않음(해당 없음). 다만 DOM을 `innerHTML` 통짜 교체하는
  패턴이 있어(목록 갱신 시) 데이터가 많아지면 비효율적일 수 있음(**정량 측정
  불가** — 실브라우저 프로파일링 필요).
- **과도한 API 호출**: 화면 전환마다 필요한 데이터만 fetch하는 구조로 보이나(뷰별
  개별 로드 함수 확인), 디바운스 없는 검색 인풋이 있다면 타이핑마다 호출될 수
  있음 — 검색은 버튼 클릭 트리거로 확인되어 이 문제는 없음.
- **Lazy Loading**: 없음(전체 스크립트가 페이지 로드 시 한 번에 실행). 지금
  파일 크기(index.html 1,839줄, 대부분 인라인 `<script>`)면 아직 체감 영향은
  적겠지만 계속 커지면 초기 파싱 비용 증가.
- **이미지 최적화**: **해당 없음** — 이미지 자산이 프로젝트에 전혀 없음(SVG를
  전부 JS로 직접 생성, `<img>` 태그 0건 확인).
- **번들 크기**: **해당 없음** — 빌드 도구가 없어 "번들"이라는 개념 자체가 없음.
  다만 index.html 파일 자체가 1,839줄(HTML+CSS+JS 전부 포함)로, 별도 파일 분리
  없이 하나의 다운로드로 전달됨 — 첫 로드 크기 최적화 여지는 있음(정확한 KB 크기는
  **확인 불가**, 이 세션에 브라우저 네트워크 탭 재현 불가).
- **캐싱**: AI 코칭 메시지는 사용자당 1일 1회 DB 캐싱(`coaching_cache` 테이블,
  코드로 확인). 그 외 HTTP 캐시 헤더(Cache-Control 등)는 정적 파일 서빙에
  명시적으로 설정된 게 안 보임(`StaticFiles` 기본 동작에 의존).
- **Pagination**: 관리자 사용자 목록에 서버사이드 페이지네이션 적용(코드 확인).
  유저 페이지 기록 목록은 **클라이언트 사이드 페이지네이션**(10건/페이지, 전체를
  한 번에 받아온 뒤 화면에서만 나눔) — 기록이 수천 건으로 늘어나면 이 방식은
  느려짐.
- **Virtualization**: 없음(리스트 길이가 아직 가상 스크롤이 필요한 규모가 아님).
- **Memoization**: 서버 쪽 캐싱(coaching_cache, badges 지연평가)은 일부 적용,
  프론트엔드 계산 결과 캐싱은 확인 안 됨.

**개선 제안**: 유저 기록 목록도 서버사이드 페이지네이션으로 전환(현재 방식은
데이터가 늘어날 미래를 대비 못 함), 정적 파일에 `Cache-Control` 헤더 추가.

## 9. 데이터베이스 설계

**테이블**: users, sessions, audit_logs, health_records, goals, badges,
coaching_cache — 7개.

**Index**: PK 전부 index=True, FK 컬럼(`user_id`) 전부 index=True, `username`
unique+index, `health_records.date` index(날짜 검색 쿼리 대응), `goals.user_id`가
unique+index로 "사용자당 목표 1건"을 **DB 레벨에서 강제**하고 있음(애플리케이션
로직뿐 아니라 제약조건으로도 보장 — 좋은 설계).

**Query**: N+1 문제를 의식적으로 회피한 흔적 확인(`admin/users`의 기록 수 집계를
`group_by` 한 번으로, 위험도 계산을 정렬 1회 조회로 처리 — PROJECT_CONTEXT.md
근거 + 코드 확인). Export/코칭 관련 쿼리는 사용자 단위로 스코프되어 있어 대량
스캔 쿼리는 없음.

**관계**: 1 User — N Session/HealthRecord/Badge, 1 User — 0~1 Goal/CoachingCache.
`audit_logs`만 의도적으로 FK 없음(계정 삭제 후에도 이력 보존 목적, 합리적 설계
판단).

**정규화**: 3정규형 수준 준수. `health_records`에 원본값과 파생값(BMI 등)을
같이 저장하는 건 엄밀히는 비정규화이지만, "계산 시점의 분류 기준으로 저장해
과거 기록의 해석이 바뀌지 않게 한다"는 의도라면 합리적 트레이드오프(단, 이
의도가 코드 주석으로 명시되어 있진 않음 — **추정**).

**확장성**: `Alembic` 도입으로 스키마 변경이 안전해짐(가장 큰 개선). 다만 SQLite
자체가 동시 쓰기에 약해, 사용자 수가 늘면 PostgreSQL 전환이 필요한 시점이 옴
(지금 규모에선 문제없음).

**개선안**: `health_records`에 `(user_id, date)` 복합 인덱스를 추가하면 "특정
사용자의 날짜 범위 조회"(가장 빈번한 쿼리 패턴)가 더 빨라짐 — 지금은 `user_id`와
`date`가 각각 단일 인덱스라 복합 조건에 최적은 아님.

## 10. API 설계

**REST 설계**: 리소스 중심 경로(`/records/{id}`, `/goals`)가 대체로 잘 지켜짐.
다만 `/health-coaching`, `/reports/weekly` 처럼 동사/복합명사가 섞인 경로도
있어 완전히 일관되진 않음.

**응답 형식**: 에러는 전역적으로 `{"detail": "..."}` 통일(직접 확인). 성공 응답은
Pydantic `response_model`로 스키마가 고정되어 있어 필드 누락/타입 불일치 위험이
낮음.

**에러 처리**: 전역 예외 핸들러가 있어 처리 안 된 예외도 500 + `{"detail": ...}`
형태로 응답(서버 다운 없이) — 확인 완료.

**Status Code**: 404/401/400/500/429/423/422/409/403 — 상황별로 적절히
분화되어 사용됨(그냥 다 400/500으로 뭉치지 않음 — 잘 되어 있는 부분).

**버전 관리**: `/v1/` 같은 API 버전 프리픽스 없음 — 지금은 단일 클라이언트(자체
프론트엔드)만 쓰므로 문제없지만, 외부에 API를 공개하거나 모바일 앱이 생기면
버전 관리 전략이 필요해짐.

**일관성/명명 규칙**: snake_case 쿼리 파라미터, RESTful 동사(GET/POST/PUT/DELETE)
사용 일관됨.

1. Unit: `health_logic.py`, `health_score.py`, `goal_prediction.py` 등 순수
   계산 로직에 대한 단위 테스트 우선 추가(외부 의존성 없어 작성 비용 낮음)
2. Integration: 나머지 라우트(AI Coach, Admin Analytics, CSV Import/Export)를
   기존 `conftest.py` 픽스처 재사용해 확장
3. E2E: Playwright를 "1회성 스크립트"가 아니라 `tests/e2e/` 디렉토리에 코드로
   고정해 회귀 스위트화 — 지금처럼 매번 새로 스크립트를 짜는 방식은 반복 검증
   비용이 계속 발생함
4. CI에서 pytest는 지금 당장 붙일 수 있음(로컬에서 이미 성공하므로), Playwright
   E2E는 헤드리스 브라우저 설치가 필요해 그다음 단계로

## 12. 배포 및 운영

**Docker**: `Dockerfile` 존재, backend/frontend 형제 폴더 구조를 유지하도록 작성됨,
로컬 빌드/실행 검증 완료(문서 근거, 이번 세션 재현은 Docker 데몬 부재로
**확인 불가**). 컨테이너 `CMD`가 기동 시 `alembic upgrade head`를 먼저 실행하도록
되어 있어(코드 확인) 배포 시 스키마 적용을 잊을 위험이 낮음 — 좋은 설계.

**CI/CD**: `.github/workflows/` 등 CI 설정 파일이 **전혀 없음**(find로 직접
확인). pytest가 로컬에서 통과하는 상태이므로 GitHub Actions 연결 자체는
당장 가능한 상태.

**환경 분리**: `.env.example`로 dev/prod 설정 항목(OPENAI_API_KEY,
ALLOWED_ORIGINS, COOKIE_SECURE)이 구분되어 있으나, "이건 dev용", "이건 prod용"
같은 별도 프로파일(`docker-compose.dev.yml` 등)은 없음 — 환경변수 하나하나를
배포 시 사람이 직접 챙겨야 하는 구조.

**로그**: `logging` 모듈 사용 확인(main.py). 구조화 로그(JSON 로그) 또는 외부
로그 수집(Sentry, CloudWatch 등) 연동은 없음.

**모니터링**: `/health` 헬스체크 엔드포인트는 있음. 알림(에러율 급증 시 알림 등)
체계는 없음.

**백업**: SQLite 파일 백업 전략(주기적 스냅샷 등)이 코드/문서 어디에도 없음 —
**배포 전 반드시 정의해야 할 항목**.

**장애 대응**: 전역 예외 핸들러로 500 격리는 되어 있으나, 장애 발생 시 알림을
받을 방법(로그 확인 외)이 없어 장애를 사람이 "우연히 발견"하기 전까지 인지하지
못할 구조.

| 부채 | 왜 부채인가 | 향후 문제 가능성 |

## 14. 시장성 분석

**주의**: 이 세션은 코드 저장소 분석에 한정되어 있어 실시간 시장 데이터(정확한
경쟁사 기능표, 사용자 수, 가격 등)는 **확인 불가**입니다. 아래는 일반적으로 알려진
건강기록 서비스 카테고리(삼성헬스/애플헬스/눔/마이핏니스팔 등) 특성과 비교한
정성적 분석입니다.

**부족한 점**: 웨어러블 실연동 부재(경쟁 서비스 대부분의 기본 기능), 소셜/커뮤니티
요소 없음, 모바일 네이티브 앱 없음(웹만 존재), 알림 없음.

**강점/차별점**: (1) AI 코칭이 규칙기반→실LLM으로 자연스럽게 전환 가능한 구조라
"AI가 실제로 분석해서 조언한다"는 메시지를 낮은 리스크로 낼 수 있음, (2) 관리자
Analytics가 이미 상당히 갖춰져 있어 B2B(기업 임직원 건강관리 등)로 피벗할 때
기반이 있음, (3) 보안/감사로그 등 "장난감 프로젝트"를 넘어선 기본기가 되어 있어
초기 신뢰 확보에 유리.

**경쟁력 평가**: 개인 프로젝트/MVP 단계 기준으로는 기능 완성도가 높은 편이나,
실제 상용 경쟁(웨어러블 연동, 모바일 앱, 커뮤니티)과 비교하면 아직 "기록+분석"
핵심에 집중된 초기 단계 제품. B2C 대중 시장보다는 **B2B(기업 건강관리 SaaS) 또는
니치(특정 만성질환 관리 커뮤니티) 포지셔닝**이 지금 자산(관리자 대시보드, 감사
로그, 위험도 지표)과 더 잘 맞을 가능성.

   만들어주지 못하면 이탈이 빠를 것
7. 나머지는 ①~③에 정리된 순서를 그대로 따르면 됩니다.

## ⑤ CTO 종합 리뷰

이 프로젝트는 **"개인 미니 과제"라는 출발점을 이미 훨씬 넘어선 상태**입니다.
인증·권한분리·감사로그·Alembic·pytest·CI 전 단계의 기본기(CI 제외)가 실제로
갖춰져 있고, 특히 매 기능마다 검증→문서화를 반복해온 개발 습관은 이 규모의
개인 프로젝트에서 보기 드뭅니다. 코드를 직접 읽어본 바, "돌아가기만 하면 된다"가
아니라 IDOR을 404로 숨기고, cascade 삭제를 실제로 DB에서 확인하고, 위험한
비밀번호를 거부하는 등 **운영을 실제로 상상하며 짠 코드**라는 인상을 받았습니다.

다만 CTO로서 인수받는다면 세 가지를 가장 먼저 짚겠습니다. 첫째, **아직 실서버에
한 번도 올라가 본 적이 없다**는 점 — 로컬에서 아무리 완벽해도 검증되지 않은
가정(성능, 실제 사용자 트래픽 패턴)이 여전히 많습니다. 둘째, **XSS 검증이 완료되지
않은 채로 상당한 범위의 innerHTML 렌더링이 존재**한다는 점 — 이건 다음 감사에서
반드시 1순위로 닫아야 할 문입니다. 셋째, **재방문을 만드는 장치(알림, 온보딩)가
없다**는 점 — 기능적으로는 완성도가 높지만, 서비스로서 "사람들이 계속 쓰게 만드는"
설계는 아직 다음 단계입니다.

**성숙도**: MVP를 넘어 "베타 서비스" 직전 단계. **상용화 가능성**: 기술적으로는
지금 배포해도 큰 무리는 없으나, 데이터 유실(백업 부재)과 보안 검증 공백을 방치한
채 실사용자 데이터를 받는 것은 권장하지 않습니다. **운영 리스크**: 단일 워커·
SQLite·백업 없음이라는 조합은 소규모 트래픽에선 무해하지만, 그대로 성장하면
가장 먼저 터질 지점입니다. **장기 확장성**: Provider 패턴·인터페이스 설계·
Alembic 도입 판단을 보면, 이 코드베이스는 "지금 이 순간"이 아니라 "다음에
무엇이 추가될지"를 의식하며 짜여 있습니다 — 장기적으로는 오히려 안심되는
방향입니다. 지금 당장의 구멍들만 순서대로 메우면, 이 프로젝트는 개인 포트폴리오
수준을 넘어 실제로 사람에게 써보라고 권할 수 있는 서비스에 가장 가까이 와 있는
상태라고 평가합니다.
